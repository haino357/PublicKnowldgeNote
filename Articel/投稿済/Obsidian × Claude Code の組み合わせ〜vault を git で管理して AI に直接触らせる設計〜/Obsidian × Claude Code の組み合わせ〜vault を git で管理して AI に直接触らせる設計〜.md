# Obsidian × Claude Code の組み合わせ〜vault を Git で管理して AI に直接触らせる設計〜

![[ObsidianClaudeCode_アイキャッチ.png]]

## 情報は溜めるより「使える状態を保つ」ほうが難しい

Obsidian を使い始めてしばらく経つと、ほぼ必ずぶつかる問題がある。

**ノートは増えるのに、古くなった情報が混ざって信用しにくくなる。**

手順書を書いた。調査結果をまとめた。業務ルールを記録した。けれど半年後に開いてみると、ツールのバージョンが変わっていたり、フローが変わっていたりして、そのまま使えるか分からない。かといって、すべての情報を手動でメンテし続けるのは現実的ではない。

この問題を解決するために、**Obsidian の vault を Git リポジトリとして管理し、Claude Code がその中を直接読み書きできる構造**を作った。

---

## 仕組みの全体像

![[ObsidianClaudeCode_全体構成図.png]]

中心にあるのは **[Karpathy](https://karpathy.ai/) の [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) パターン**という考え方だ。「原典・受信箱・wiki・完成品・アーカイブ」を分離して管理することで、情報が混在しない設計にする。

```
my-vault/（Obsidian vault = Git リポジトリ）
│
├── 000.raw/         ← 外部原典の不変保管庫（Web クリップ、公式資料）
├── 001.Daily/       ← 日次記録（append-only）
├── 100.inbox/       ← 未仕分けの一時バッファ
├── 110.wiki/        ← 知識ベース（技術手順・業務知識）
├── 120.output/      ← 完成品・納品物
├── 200.projects/    ← プロジェクト・業務別資料
└── 999.archive/     ← 完了・廃止コンテンツの凍結
```

**Obsidian でノートを書き、Claude Code が vault 内を直接操作する。** テキストエディタと AI が同じファイルを扱うので、「AI に伝えるための別ツール」は不要になる。

---

## Obsidian Web Clipper との連携

![[ObsidianClaudeCode_WebClipper連携図.png]]

ブラウザで気になった記事や公式ドキュメントは、Obsidian Web Clipper で原典保存用のフォルダに保存する。

このフォルダは `.gitignore` でクリップファイル自体を Git 管理外にしている。理由は単純で、Web クリップは外部コンテンツであり、個人の閲覧履歴にも近い。大量に溜まっても、Git で差分を追跡する価値は薄いからだ。リポジトリにはフォルダ構造だけを保持している。

`000.raw/` に入ったものは Claude Code も**読むだけで編集しない**というルールにしている。原典の保護が最優先だ。必要なときに読んで、`110.wiki/` のページを書く素材にする。

## 「slash commands」で日常操作を統一

![[ObsidianClaudeCode_slashcommands日次フロー.png]]

Claude Code では、コマンド定義用のディレクトリにファイルを置くと `/コマンド名` で呼び出せる。よく使う操作をコマンド化した。

| コマンド | タイミング | やること |
|---|---|---|
| `/start` | 始業 | デイリーノート作成・前日の繰り越し・inbox 件数・優先度つきIssueを一覧表示 |
| `/sort` | inbox が溜まったとき | ファイルを読んで分類先を提案・承認後に移動 |
| `/done` | 終業時 | 会話の文脈から「今日やったこと」を自動抽出・デイリーノートに追記 |
| `/lint` | 定期的に | wiki 全体をスキャンして古いページ・壊れたリンクを検出 |
| `/stage` | コミット前 | ステージング＋コミットメッセージ案を同時に提示 |

特に便利なのが `/done` と `/lint` だ。

**`/done`** は終業前に実行すると、その日の会話の中で行ったファイル操作、調査、設計判断などを自動で拾い、デイリーノートへの追記候補を出してくれる。自分で振り返りを書く手間がかなり減った。

**`/lint`** は `110.wiki/` 全体をスキャンして「3ヶ月以上更新されていないページ」「壊れたリンク」「フロントマターの欠損」を検出する。wiki の品質維持を自動化できる。

## 設計上の重要な判断：何を自動更新して、何を手動にするか

![[ObsidianClaudeCode_自動更新線引き図.png]]

この仕組みを作るうえで一番悩んだのが、**「どこまで自動更新するか」** の線引きだ。

結論は次の通り。

- **仕事の情報**（業務ルール、案件情報、手順書）→ 鮮度が命なので**積極的に自動更新**
- **個人のメモ・日次記録**→ 当時の思考や判断を残すものなので**自動で書き換えない**

日次記録を自動で書き換えてしまうと、「なぜその日そう考えたか」という記録としての価値が失われる。一方、業務手順書が半年前のバージョンのまま放置されていると実害が出る。

Claude Code への指示もこの方針に合わせた。プロジェクト資料や業務ナレッジでは、更新提案を積極的に出す。一方で、日次記録は `/done` のような append-only コマンド経由以外では触らない。

---

## 会話をまたいだ記憶（自動メモリ）

Claude Code には、会話間でユーザーの行動ルールや嗜好を記憶する「自動メモリ」機能がある。ローカルのメモリ用ディレクトリに Markdown ファイルとして蓄積される。

たとえば：
- 「`git add` したら同じ返答でコミットメッセージ案も必ず出す」
- 「`/done` の追記は作業単位ごとに見出し＋開始・終了時刻を付ける」
- 「vault 内の新規 `.md` は必ず Obsidian Flavored Markdown で書く」

これらを一度指摘しておけば、次の会話からも同じルールを守りやすくなる。毎回「こういうスタイルで」と伝える必要がない。

## まとめ

![[ObsidianClaudeCode_鮮度維持システム変換図.png]]

Obsidian × Claude Code の組み合わせで実現できたのは、**「情報を溜めるシステム」から「情報の鮮度を保つシステム」への変換**だ。

- Web Clipper で外部情報を `000.raw/` に収集
- `/lint` で定期的に wiki の品質をチェック
- 仕事情報は自動更新、個人メモは手動維持という方針で管理
- slash commands で日常操作を統一し、Claude Code が vault を直接扱う

特別なサーバーや専用ツールは不要だ。Obsidian の vault を Git で管理して、Claude Code をその上で動かすだけで構築できる。

情報管理に悩んでいる Obsidian ユーザーや、Claude Code の活用方法を探している人の参考になればうれしい。
